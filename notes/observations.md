## 2026-03-26 — Done-task double response + wrong tone

**Observed:**
```
[23:42] gm: done Get rid of Substack account
[23:42] eva: Done. What's next, Gabriell?
[23:42] eva: Nice work, Gabriell! Marking that done. 🎉
```

**Root cause — double response:**
Two messages were sent, not one. Discord shows both at [23:42].
- Message 1: `done` → FINISHED intent → hardcoded "Done. What's next, Gabriell?"
- Message 2: `Get rid of Substack account` → GENERAL intent → LLM sees the previous "done"
  exchange in interaction history, infers task completion, responds enthusiastically.

The UX friction: the user's natural phrasing is `done <task text>` (no colon), but only
`done: <task text>` (with colon) routes to DONE_TASK. The colon is too easy to forget.

**Root cause — wrong tone:**
The LLM response "Nice work, Gabriell! Marking that done. 🎉" is being generated by the
GENERAL handler without a tone constraint. Also the FINISHED handler's hardcoded
"Done. What's next?" is passive — it should push for one more action, not ask.

**Fixes:**

### Fix 1 — Accept `done <task text>` without colon as DONE_TASK

In `detect_intent`, extend DONE_TASK detection to match `done <text>` when the text after
"done" is not a preposition/filler phrase:
- `done: fix bug` → DONE_TASK (existing)
- `done fix bug` → DONE_TASK (new)
- `done` → FINISHED (standalone, no change)
- `done with the slides` → FINISHED (preposition guard)
- `done already`, `done it`, `done for today` → FINISHED (preposition guard)

Update `_handle_done_task` to strip both `done: ` and `done ` prefixes.

### Fix 2 — Harden FINISHED response (time-aware, hardcoded)

Replace `"Done. What's next, Gabriell?"` with a time-aware push:
- After 20:30 (recovery): "Good. One more before you close out — what's it going to be?
  The version of you that wins acts now."
- Before 20:30 (work/general): "Good. Keep moving. What's next on the list?
  The version of you that succeeds acts immediately."

No LLM call needed — hardcoded variants keep it fast and zero-cost.

**Status:** Fixed (see commit after this note).
